Zephyr project RTOS

Introduction of Zephyr

Zephyr is an open-source real-time operating system (RTOS) designed to be scalable, secure, and optimized for resource-constrained devices. It is maintained by the Linux Foundation and has a growing community of developers.

Zephyr's design aims to provide a lightweight and modular kernel that can be configured for various use cases, from small microcontrollers to more powerful microprocessors. It supports multiple architectures, including ARM, x86, RISC-V, and others.

Zephyr provides a rich set of features, including preemptive multitasking, real-time scheduling, memory protection, device drivers, and networking support. It also offers a variety of application-level libraries and protocols, such as Bluetooth, Wi-Fi, USB, and MQTT.

Zephyr is licensed under the Apache License 2.0, which allows for both commercial and non-commercial use, modification, and distribution. The project has a growing community of developers and contributors who are working on improving and expanding its capabilities.

Device Tree Source

In computing, a devicetree (also written device tree) is a data structure describing the hardware components of a particular computer so that the operating system's kernel can use and manage those components, including the CPU or CPUs, the memory, the buses and the integrated peripherals.

The device tree was derived from SPARC-based computers via the Open Firmware project. The current Devicetree specification is targeted at smaller systems, but is still used with some server-class systems (for instance, those described by the Power Architecture Platform Reference).

Personal computers with the x86 architecture generally do not use device trees, relying instead on various auto configuration protocols (e.g. ACPI) to discover hardware. Systems which use device trees usually pass a static device tree (perhaps stored in EEPROM, or stored in NAND device like eUFS) to the operating system, but can also generate a device tree in the early stages of booting. As an example, Das U-Boot and kexec can pass a device tree when launching a new operating system. On systems with a boot loader that does not support device trees, a static device tree may be installed along with the operating system; the Linux kernel supports this approach.

The Devicetree specification is currently managed by a community named devicetree.org, which is associated with, among others, Linaro and Arm.

    ...

Devicetree node requirements

Includes child node for eeprom from specific i2c device parent node:

  &i2c_smb_0 {
  	...
  	eeprom0: eeprom0@50 {
  		compatible = "atmel,at24";
  		status = "okay";
  		reg = <0x50>;
  		size = <4096>;
  		address-width = <12>;
  		timeout = <25>;
  		pagesize = <32>;
  	};
  }

As example above for AT24C32 which is 256 pages of 32 bytes each.

The parent node i2c_smb_0 named by microchip soc family as i2c controller 0, extends eeprom0 as child node under i2c_smb_0.

Reference device binding from eeprom device at24 from vendor atmel, as compatible = "atmel, at24", all node properties compatible reference from atmel, at24.yaml.

The status = "okay" to enable this eeprom0 child node on i2c interface eeprom features, which initialization on eeprom_at2x.c.

Every I2C interface should contains target address included this 2 wire serial EEPROM, which target address depends on EEPROM pins (A2, A1, A0) with fixed most signification nibble 0x50. As above target address is selected as 0x50 reg = <0x50>.

EEPROM AT24C32 is 128 pages of 32 bytes which is : 4096 Bytes size = <4096>.So, the address width for 4096 (0x10000) bytes is 12 bits address-width = <12>

Reply access eeprom until success within 25ms, depends on timeout = <25>

Every EEPROM will supporting page write mode, for AT24C32 is 32-byte page write mode pagesize = <32>, which prevent 'roll-over` on write over 32 bytes, previous data will be overwritten.
